### Bragi photo-vision CATALOG/CACHE with HIERARCHICAL / FACETED tagging (optimization - build AFTER vision selection works)

Problem: re-running AI vision on every CompanyCam photo for every article is expensive and slow. It pays the vision cost on the same photo over and over, N times per photo across N articles. The better model is to remember what each photo already contains instead of re-flipping the entire deck every time.

Core fix - persistent catalog DB: when AI vision analyzes a photo once, store the result as `photo ID -> content tags`. Future articles query the catalog and go straight to relevant, already-tagged photos. Only new or unseen photos get vision-analyzed and then added to the catalog. Each photo is analyzed once, not once per article. Savings compound as the library gets catalogued, and generation gets faster as coverage grows.

Key design - tags must be hierarchical / faceted, not flat. Use Chris's car-dealership-funnel insight: do not tag a photo with one flat label like `vinyl liner`. Tag it across multiple independent facets, each able to narrow from general to specific so matching behaves like a sales funnel. Example mental model: vehicle -> truck -> full-size -> 4-door -> red -> Chevrolet -> Colorado. Each answer shrinks the pile.

Suggested facet dimensions to refine at build time:
- Subject: pool overview / underwater close-up / filtration equipment / filter room / broken plumbing / excavation for repair
- Pool type: vinyl liner / gunite / fiberglass
- Attributes: color (for example blue), pattern (for example shell pattern), and similar descriptive traits
- Problem shown: leak at seam / cracked fitting / liner tear / equipment failure / water loss in filter room

Bragi should match an article on whichever facets matter for that specific article, at the most specific level available, and then fall back gracefully up the hierarchy when there are not enough precise matches. Example: `blue vinyl shell-pattern seam leak` -> if none, widen to `blue vinyl liner leak` -> then `vinyl liner leak`. The goal is precision when the library supports it and graceful degradation when it does not. Location is not a facet; this catalog is content-only.

Trigger to build: after Mode B's AI-vision photo selection is proven working. Do not build a cache/catalog around an unproven selection approach. Principle: make it work, then make it fast. This is a v2 optimization layer on top of working vision selection.

### Mode B - Reply-based publish/schedule approval (CHOSEN approach, build AFTER Mode B works)

When a Mode B draft is ready, the trimmed review email (now Title + Draft URL only) should also let Chris approve publishing or scheduling by replying to the email.

Chosen approach: reply-based approval over signed one-click links or a dashboard because it carries the lowest accidental-publish risk to the live business site and reuses the existing approval plumbing already built in `bragiTelegramApprovalService.js` (pending-draft + publish/schedule pattern).

Natural-language intent, not rigid keywords: Clawdia must understand the meaning of the reply instead of requiring exact command words.

Examples:
- `it's good, let's publish` -> publish now
- `yes I like it, schedule it for April 3 2026 at 2:38pm` -> schedule for that exact date/time
- `needs work, revise the intro` -> revise / hold
- `park this one` -> park

Parse intent as publish / schedule + datetime / revise / park from plain language, including extracting the date and time for scheduling.

Confirmation handshake is required and is also a safety mechanism: after parsing Chris's reply, Clawdia must reply back confirming what she understood and what she is doing, for example `Got it - publishing '[Article Title]' now` or `Scheduled '[Title]' for April 3, 2026 at 2:38 PM.` This proves the reply was received and executed, and gives Chris a chance to catch a misread before or at execution. For publish specifically, because it is irreversible on the live site, the confirmation should give a beat to stop if she misread intent.

Security requirements: exact sender match, per-draft nonce/token, and pending-state match. Human gate always; never auto-publish. Natural-language parsing must not loosen the security match. It still has to verify the reply genuinely came from Chris and is tied to the correct pending draft.

Trigger to build: only after Mode B is actually producing real AI-generated articles and those articles are proven good. Do not build publish approval on top of a content engine that is not yet generating real content.

Long-term alternative if volume grows: a small approval queue/dashboard. Safest, but more effort. Revisit only at scale.

## CompanyCam - Library Scale & Cheap-Tagging Path

### Audited 2026-06-27, live API, read-only

### Library scale (live counts)

- Total photos: 22,669
- Total videos: 1,103
- Distinct projects with photos (historical, from photo feed): 954
- Note: live `GET /v2/projects` only returned 50 current projects, but the photo feed showed 954 distinct `project_id` values, so 954 is the real `projects with at least one photo` number.
- Implication: 22,669 photos is large. Running AI vision on all of them would be a real cost. That is exactly why the cheap-tagging path below matters - minimize vision, maximize use of data already in the account.

### Confirmed feasible: cheap-tagging hierarchy (use existing data, vision only for gaps)

CompanyCam API capabilities verified live and against docs:

- Tags: API can write multiple custom tags per photo (`POST /v2/photos/{id}/tags`), read tags (`GET /v2/photos/{id}/tags`), and list/create custom tags (`GET/POST /v2/tags`). Multiple tags per photo are supported.
- Address/geo: free and exact. `photo -> project_id -> GET /v2/projects/{id}` returns full address (`street`, `city`, `state`, `postal_code`) plus photo coordinates (`lat`, `lon`). No AI is needed for location.
- Documents: `GET /v2/projects/{project_id}/documents` lists docs; PDF reports are downloadable (proven live: `200`, `application/pdf`). Reading PDF text is on NexTeam's side, not an API blocker. A PDF parser is still needed and was not yet added at audit time.
- Linkage proven live: `photo -> project -> address -> documents` all connect.

### The cheap-tagging stack (the cost-saver)

1. Project address = free, exact geo tags with no AI.
2. Project PDF/report text = read findings such as plumbing, filter, liner, skimmer, light, and pool type into project-context tags using cheap text extraction instead of expensive vision.
3. Write those tags back into CompanyCam and/or the local catalog.
4. Use AI vision only as a fallback for photo-specific subject judgment, such as deciding which exact image is the underwater close-up, equipment-pad shot, or overview.

### Catches (honest)

- Reports describe the project, not which specific photo is which. A project can contain leak shots, equipment shots, overview shots, and context photos. So document-derived tags are strong project-context tags, not a full replacement for photo-subject tagging. Vision still fills the photo-specific gap, but only the gap - not all 22,669 photos.
- `GET /v2/photos` does not include tag arrays inline. Reading tags photo-by-photo across 22k photos would balloon API calls, which makes the local catalog/cache (`companycam-photo-catalog.v1.json`) essential, not optional.

### Catalog readiness

- The faceted catalog in `bragiModeBPhotoCatalog.js` already has `projectContext` (`projectId`, address snapshot, document IDs/names, derived tags, report signals) and `textSignals` fields, so the cheap-tagging data already has a home.
