### Building-analogy discipline

- Blueprint before building, source materials before bidding, pre-stage materials for the fine-tune: correct.
- But do not order materials for all 10 floors at once. Order for the floor being built now plus the next one.
- The trap, especially with ADHD and fear of being late from missing .com/Bitcoin, is sourcing all floors at once and ending with many half-built floors and zero finished.
- Early-but-scattered loses to focused-and-shipping.

### The one floor that proves the building

- Bragi working on Aquatrace end-to-end, producing real content that makes Aquatrace more findable.
- Ship that one floor a customer can stand on.
- It proves NexTeam is real, not just an idea.
- It gives Chris something to show a second client.
- It creates the "solid, revenue, brand-building, active quickly" outcome he wants.
- He is close. Mode B is roughly one real test run away. Weeks, not years, if he stays on it.

## Load-bearing build rule

- Every floor should be built to hold future load, not just today's demo.
- Every system adds weight: memory, token/API usage, storage, compute, and recurring cost.
- Before building or endorsing a path, ask how much weight it adds today and what happens when 1,000+ clients are using it.
- Ask whether the design is reusable across clients or only works because of one-off hardcoding.
- Ask what resource limits or cost ceilings exist and whether the design respects them.
- If a fast path is only a deliberate v1 shortcut, label it honestly and state the scale-up path instead of pretending it is final architecture.

## The decision filter

Use this on every "should we build X?" question:

1. Does this help finish the current floor (Bragi / Mode B working on Aquatrace), or am I sourcing materials for floor 7 out of fear of being late?
2. Does it serve the real product vision: one engine that generates business, not a disconnected tool pile?
3. Does it create revenue that does not cost Chris's time?
4. Is it reusable/config-driven so Aquatrace work becomes future-client infrastructure?
5. What weight does it add in memory, token/API usage, storage, compute, and recurring cost today and at 1,000+ clients?
6. Is it free/logic prep that can be staged-and-waiting, or paid/decision work that needs Chris present?
7. If it is a v1 shortcut, is the scale-up path named explicitly?

Park good ideas with triggers. Finish one working floor before opening new avenues.
